# ADR 012 тАФ Gold row-level security

## Status
Accepted (supersedes earlier `plant_access_filter` approach; refines ADR 005)

## Context
ADR 005 made the Gold layer a **trusted aggregate**: attaching a UC row filter directly to a
materialized view forces **full MV refreshes** (cost + latency), so Gold MVs stay unfiltered.
The gap: anything granted direct SELECT on a Gold table sees all plants.

Silver row filters were also dropped: the `silver_io_reporting` schema is not granted to the
consumer group (`users`) тАФ civilians have no USAGE or SELECT on it. Schema-level access control
is sufficient for silver; per-row filtering there is redundant overhead.

## Decision

### 1. Gold MVs тЖТ secured serving views
For each plant-scoped Gold table, expose a `<table>_secured` view that enforces plant access
via the **central `published_<env>.security.model` CSM pattern**:

```sql
CREATE OR REPLACE VIEW <catalog>.gold_io_reporting.<table>_secured AS
  SELECT * FROM <catalog>.gold_io_reporting.<table>
  WHERE EXISTS (
    SELECT 1 FROM published_<env>.security.model
    WHERE current_user() = email
      AND application_key = 'io_reporting'
      AND LOWER(access_type) = 'full view'
    UNION ALL
    SELECT 1 FROM published_<env>.security.model
    WHERE current_user() = email
      AND application_key = 'io_reporting'
      AND LOWER(access_type) = 'filter'
      AND array_contains(filter_plant, plant_code)
  );
```

- `access_type = 'full view'` тЖТ user sees all plants (admins / cross-plant roles).
- `access_type = 'filter'` тЖТ user sees only the plant codes in their `filter_plant` array.
- Access is managed centrally in `published_<env>.security.model` by the platform security team.
  Users need only one onboarding entry across all connected-plant applications.
- Dev secured views are **pass-through** (no filter) тАФ no `published_dev.security` exists and dev
  is not a consumer-facing environment.

Generated by `scripts/generate_gold_security_sql.py` тЖТ `resources/sql/gold_security_<env>.sql`.
Run once per env by a UC admin after deploy (re-runnable; views are `CREATE OR REPLACE`).

### 2. Consumption moves to the views
The `users` group is granted SELECT on the `*_secured` views; direct SELECT on the base Gold
tables is reserved for the data-product owner / admins. The hardening REVOKE script
(`resources/sql/gold_security_harden_<env>.sql`) is applied separately (operationally sensitive).

**Order matters, and is now verified.** The two scripts must be run in order
(`gold_security_<env>.sql` тЖТ `gold_security_harden_<env>.sql`); a skipped harden step leaves base
Gold tables readable by `users`. The **`gold_security_verify` job** (`resources/gold_security_job.job.yml`,
driver `gold/security/verify_gold_security.py`) automates the verification gate: it discovers every
`*_secured` view and asserts the `users` group has SELECT on the secured view but **not** on its base
table, failing when `--enforce=true`. It defaults to verification-only / report mode (`--enforce=false`);
flip to enforce once UAT is clean. It can also apply the two scripts in order (`--apply=true`), but the
priority тАФ and default тАФ is the verification gate, so the manual admin run above remains the apply path
until the apply mode is exercised. Verification core unit-tested in `tests/test_verify_gold_security.py`.

### 3. Snapshot tables тАФ no row filter
`gold/snapshots/warehouse_snapshot.py` writes `*_snapshot` companion tables. These no longer
carry a UC row filter. Access control is the same as Gold MVs: consumers query the
`*_secured` views, not the base snapshot tables directly.

### 4. Future org tiers (ADR 005 cluster-lead)
The `published_<env>.security.model` table already carries `filter_cluster`, `filter_region`,
`filter_country`, and other org-hierarchy columns. Adding a cluster-lead tier requires only
adding an `OR array_contains(filter_cluster, cluster_code)` branch to the EXISTS check тАФ no
new governed seed tables or UC attribute plumbing.

## Consequences
- The secured-view SQL must be re-applied when Gold tables are added/renamed (keep `GOLD_TABLES`
  in `scripts/generate_gold_security_sql.py` in sync).
- `published_prod.security.model` must exist and contain `io_reporting` rows before prod
  consumers can query secured views (tracked in `docs/ingestion_requests.md`).
- Dev secured views return all rows тАФ appropriate for a non-consumer-facing environment.
- Silver schema access control (`GRANT USAGE / SELECT` on `silver_io_reporting`) is managed
  separately at the UC catalog/schema level; no row-filter SQL is required.
