# ADR 012 — Gold & snapshot row-level security

## Status
Accepted (refines ADR 005)

## Context
ADR 005 made the Gold layer a **trusted aggregate**: plant row filters are applied on Silver,
not on the Gold materialized views, because attaching a UC row filter to a materialized view
forces **full MV refreshes** (cost + latency). The gap: anything granted **direct** SELECT on a
Gold table — and especially on the append-only `*_snapshot` tables, which accumulate historical
plant data — sees **all plants**. For a multi-plant operational product that is too weak if
plant-scoped users ever get Gold/snapshot access.

The two cases have different physics:
- **Gold MVs** — filtering them directly is what triggers the refresh cost ADR 005 avoids.
- **Snapshot tables** — physical Delta tables written by a batch job, *not* MVs, so a real row
  filter costs nothing extra.

## Decision
1. **Gold MVs → secured serving views.** For each plant-scoped Gold table, expose a
   `<table>_secured` view: `SELECT * FROM <table> WHERE silver.plant_access_filter(plant_code)`.
   The MV stays trusted/incremental (no refresh cost); the view does per-plant trimming at query
   time (`plant_access_filter` reads the *invoking* user's `allowed_plants` attribute; `silver_admin`
   bypasses). Generated by `scripts/generate_gold_security_sql.py` →
   `resources/sql/gold_security_<env>.sql`, applied once per env by a UC admin after deploy.
2. **Consumption moves to the views.** The `users` group is granted SELECT on the `*_secured`
   views; direct SELECT on the base Gold tables is reserved for the data-product owner / admins
   (an optional `REVOKE` block is included). This is a real change for BI/queries — point them at
   `*_secured`.
3. **Snapshot tables → real in-job row filter.** `gold/snapshots/warehouse_snapshot.py` applies
   `ALTER TABLE <snap> SET ROW FILTER silver.plant_access_filter ON (plant_code)` after each
   write/retention pass (idempotent). Wired via the job's `--row_filter_function` parameter.

## Consequences
- **Run-as principal must be in `silver_admin`.** Once a snapshot table is row-filtered, the
  snapshot job's own idempotency/retention `DELETE`s read through that filter — a non-admin
  principal would only see its own plant rows and silently mis-maintain history. The function's
  `silver_admin` bypass is the intended escape hatch; the principal also needs EXECUTE on the
  function and MODIFY on the tables. (Tracked in `docs/ingestion_requests.md`.)
- The secured-view SQL must be re-applied when Gold tables are added/renamed (keep
  `GOLD_TABLES` in the generator in sync). If the Gold schema moves to a dedicated namespace
  (the open `gold` collision decision), update `gold_schema` in the generator + the row-filter
  generator accordingly.
- Local/dev runs without UC pass no `--row_filter_function`, so the snapshot job skips filtering
  (and tests run on `spark_catalog`, which does not support row filters).
